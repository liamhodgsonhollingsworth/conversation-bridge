# Conversation Bridge Protocol (CBP) v1

A minimal open protocol for receiving conversation captures from the
[Conversation Bridge](../README.md) browser extension — or any other client
that wants to relay conversations from a webpage to a user-configured
endpoint with a transparency-based trust handshake.

This document is the source of truth. The current protocol version is **1**.

---

## 1. Goals

1. **Transparency-first.** A user can read this spec, install the extension,
   point it at their endpoint, and understand exactly what will flow where.
2. **General-purpose.** Nothing in here is specific to claude.ai, Alethea,
   WeaveMind, or any other downstream consumer. Any project can host a
   compatible endpoint.
3. **Minimal surface.** Three endpoints, one event shape. Implementations
   should fit in one file in any language.
4. **Optional auth, optional signing.** Localhost endpoints can skip both.
   Public endpoints layer on a bearer token and (optionally) HMAC signatures.

## 2. Endpoint contract

A CBP-compatible endpoint exposes three HTTP routes.

### 2.1 `GET /spec`

Returns the endpoint's self-description. The extension fetches this BEFORE
saving a new connection and displays the contents to the user as part of the
trust handshake.

Response body (JSON):

```jsonc
{
  "name": "string, required — human-readable endpoint name (e.g. 'My local bridge')",
  "version": "string, required — endpoint software version (e.g. '0.1.0')",
  "accepts": ["conversation.complete", "conversation.captured", "manual.push"],
  "dataFlow": "string, required — plain-language description of what the endpoint does with received data (e.g. 'Writes each event as JSON to ~/.bridge-inbox/<eventId>.json on the local disk. Files are kept until manually deleted. No network egress.').",
  "contact": "string, optional — email or URL",
  "repository": "string, optional — source code URL"
}
```

The `dataFlow` field is load-bearing. It is what the user is consenting to.
Honest endpoints disclose what they do. Implementers SHOULD keep this field
accurate as their behavior evolves; the extension hashes the spec and
re-prompts the user if it changes.

### 2.2 `GET /health`

A lightweight liveness check. Returns:

```json
{ "status": "ready" }
```

`status` is one of `"ready"`, `"degraded"`, or `"down"`. Endpoints should
respond quickly (target: <100ms) and avoid expensive side effects. The
extension polls this on a low-frequency timer to surface broken connections
in the popup.

### 2.3 `POST /events`

Accepts a single event. Body shape (JSON):

```jsonc
{
  "id": "string, required — UUID v4 generated by the client",
  "type": "conversation.complete | conversation.captured | manual.push",
  "timestamp": "ISO 8601 string, required",
  "sourceUrl": "string, required — the URL the event was captured from",
  "sourceName": "string, optional — friendly source label",
  "payload": { /* see §3 */ },
  "metadata": { /* free-form, e.g. user agent, privacy mode */ },
  "signature": "string, optional — see §4.2"
}
```

Response on success (200):

```json
{ "accepted": true, "eventId": "<id>" }
```

Response on rejection (4xx):

```json
{ "accepted": false, "reason": "human-readable string" }
```

## 3. Payload shapes

The `payload` field's structure depends on `type`.

### `conversation.complete` and `conversation.captured`

```jsonc
{
  "messages": [
    { "role": "user" | "assistant" | "system", "content": "string", "timestamp": "string, optional" }
  ],
  "title": "string, optional",
  "url": "string, optional",
  "shareUrl": "string | null, optional — set when the source URL is a share link"
}
```

### `manual.push`

Same shape as `conversation.complete`. The semantic difference is intent —
`manual.push` means the user explicitly clicked "Push" in the popup,
`conversation.complete` means the client decided the conversation had
reached a milestone (e.g. share link generated, approval keyword detected).

## 4. Authentication and signing

### 4.1 Bearer token (optional)

If the user configured an `authToken` for the connection, the extension
sends:

```
Authorization: Bearer <token>
```

Endpoints SHOULD validate this header and respond `401` if absent or
invalid. Endpoints MAY accept unauthenticated requests if they are
listening on localhost only — the typical case for a personal bridge.

### 4.2 HMAC signature (optional)

If the user configured an `hmacSecret` for the connection, the extension
includes a `signature` field in the event body. The signature is the
hex-encoded HMAC-SHA256 of the JSON body (computed BEFORE the signature
field is added, then re-added). Endpoints validate by recomputing.

This is useful for cross-internet endpoints where transit might be modified.
Localhost endpoints typically skip this.

## 5. Trust handshake (client-side)

The Conversation Bridge extension implements the following flow when the
user adds a connection. Other clients SHOULD implement an equivalent flow:

1. User enters endpoint URL, optional token, scope patterns, event types.
2. Client calls `GET /spec`. If unreachable, the connection cannot be saved.
3. Client displays:
   - The endpoint's self-declared `name`, `version`, `repository`, `contact`.
   - The endpoint's `dataFlow` text verbatim.
   - A summary of what THIS connection will send: which scopes, which
     events, whether a token is attached, whether the payload includes
     content or is metadata-only (privacy mode).
4. User must click "Accept and enable" to persist. The client stores the
   SHA-256 of the spec; if the spec ever changes, the client SHOULD re-prompt.

## 6. Revocation

Revocation is unilaterally client-side. The user deletes the connection in
the popup; no further events are sent. There is no server-side token
revocation in v1 (endpoints can rotate their own validation if needed).

## 7. Versioning

This document specifies version **1**. Future versions will be additive
where possible. Endpoints SHOULD include their supported version in
`/spec`'s top-level fields if and when v2 ships.
